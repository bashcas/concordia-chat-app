# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

Concordia is a Discord-like chat platform built as a **polyglot microservices monorepo**. Five languages coexist with five separate build systems rooted at the repo root:

- `go.work` — Go services (`gateway`, `presence`, `audit`) plus shared Go libs under `contracts/lib/go/`
- `Cargo.toml` — Rust workspace (`services/chat`, `contracts/lib/rust/authmw`)
- `pom.xml` — Maven parent POM aggregating `services/auth`, `services/servers`, and `contracts/lib/java/*`
- `package.json` — npm workspaces for `apps/*` (Next.js web app + Electron desktop shell)
- `services/voice`, `services/tips` — Python (FastAPI), no root manifest; per-service `requirements.txt`

When you touch a shared contract, the consumers in every language must be regenerated/rebuilt — see `contracts/proto/generate.sh` for proto stubs (output to `contracts/proto/gen/{go,java}`); Java gRPC service stubs are generated by the `servers` Maven build, not the script.

## Running the stack

Always go through `./infra/up.sh` rather than raw `docker compose`. It reads `infra/.env`, validates the security flag combination, sets `COMPOSE_PROFILES`, derives `GATEWAY_TLS_ENABLED` / `GATEWAY_ALLOWED_ORIGINS` / `NEXT_PUBLIC_API_URL`, and picks the right compose overlay:

- `./infra/up.sh --build` — start the stack
- `./infra/up.sh --print` — dry-run; show the resolved mode and exit
- `./infra/up.sh -d` — detached
- Direct mode (`SECURITY_REVERSE_PROXY=false`) layers in `infra/docker-compose.gateway-direct.yml`

**Invalid combo `up.sh` rejects:** `SECURITY_REVERSE_PROXY=false` with `SECURITY_NETWORK_SEGMENTATION=true` — the gateway isn't published to the host when the private network is internal.

`infra/.env` is git-ignored; copy from `infra/.env.example`. `JWT_SECRET` must be ≥32 chars.

## Per-language test/run commands

```bash
# Go
go test ./services/gateway/...
go test ./services/presence/...
go test ./services/audit/...

# Java (run from repo root — parent POM aggregates modules)
mvn test -pl services/auth
mvn test -pl services/servers

# Rust
cd services/chat && cargo test

# Python
cd services/voice && pytest
cd services/tips  && pytest

# Web app (Next.js + Jest + Playwright)
npm test --workspace=apps/web-app
npm --workspace=apps/web-app run lint
npx playwright test --config apps/web-app/playwright.config.ts
```

Top-level npm scripts: `npm run dev:web`, `npm run build:web`, `npm run dev:desktop`, `npm run build:desktop`. The desktop app pins `NEXT_PUBLIC_API_URL=http://localhost:8080`, so it needs direct gateway mode (`SECURITY_REVERSE_PROXY=false`, `SECURITY_NETWORK_SEGMENTATION=false`).

## Architecture invariants

**Single public ingress.** With default flags only the Nginx reverse proxy (`services/reverse-proxy/`) publishes host ports (80/443, plus 8088 for the plain-HTTP listener used by Cloudflare tunnels). Every other container is on the `private-net` Docker network (`internal: true`) and is only reachable via `docker compose exec`. Never assume `localhost:<service-port>` works from the host — it only does in direct/debug mode.

**JWT propagation.** The gateway validates the JWT and forwards `X-User-ID` (and related headers) to upstreams. Downstream services trust those headers and must not be reachable except through the gateway — this is why segmentation matters. Auth/JWT logic shared across languages lives in `contracts/lib/{go,java,rust,python}/authmw`.

**Permission checks via gRPC.** Before chat/voice/etc. operate on a channel, callers invoke `PermService.CheckPerm` (`contracts/proto/check_perm.proto`) against the Servers service on port `50051` (internal-only). New endpoints that touch channels should go through this same check.

**Audit trail is event-driven.** Producers (Auth, Servers, Chat, Voice, Gateway) emit security events to the Kafka `audit.events` topic. The Audit service (`services/audit/`) hash-chains them into `audit-db` for tamper evidence. With `SECURITY_AUDIT_TRAIL=false` the consumer and DB are absent but **producers still emit** — verify with `docker compose exec kafka kafka-console-consumer.sh --topic audit.events --bootstrap-server kafka:9093 --from-beginning`.

**SSR is part of the private network.** The Next.js web app is SSR, so the Node server that renders pages lives on `private-net` and calls the gateway internally. Only the browser is "the client." Browser-side fetches use the origin-relative `NEXT_PUBLIC_API_URL=/api` so the app works on `localhost`, a LAN IP, or a Cloudflare tunnel without a rebuild. See `docs/security_patterns_implementation_v3.md` §1.3.

## Shared contracts

`/contracts` is the single source of truth — never let a service define its own copy:

- `contracts/proto/check_perm.proto` (+ generated stubs in `gen/go`, `gen/java`)
- `contracts/kafka-schemas/*.json` — JSON Schema for `user-registered`, `message-created`, `mention`, `audit.events`
- `contracts/openapi/gateway.yaml` — REST contract for the gateway
- `contracts/lib/{go,java,rust,python}/{authmw,audit}` — shared JWT middleware and audit-emit helpers consumed by services in their native language
